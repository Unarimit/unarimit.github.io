# Effective C#读书笔记

## 按章节记录
### 第一章　C#语言的编程习惯（Language Idioms）
1. 优先使用隐式类型的局部变量
2. 考虑用readonly代替const
3. 优先考虑is或as运算符，尽量少用强制类型转换
4. 用内插字符串取代string.Format()
5. 用FormattableString取代专门为特定区域而写的字符串
6. 不要用表示符号名称的硬字符串来调用 API
7. 用委托表示回调
8. 用null条件运算符调用事件处理程序
9. 尽量避免装箱与取消装箱这两种操作
10. 只有在应对新版基类与现有子类之间的冲突时才应该使用 new修饰符

### 第二章　.NET的资源管理（.NET Resource Management）
11. 理解并善用 .NET的资源管理机制
    - 在？[看看GC](../Language/GC) 

12. 声明字段时，尽量直接为其设定初始值
    - 减少重复赋值

13. 用适当的方式初始化类中的静态成员
14. 尽量删减重复的初始化逻辑
15. 不要创建无谓的对象
16. 绝对不要在构造函数里面调用虚函数
17. 实现标准的dispose 模式

### 第三章 合理地运用泛型（Working with Generics）

泛型类和泛型方法，感觉在业务开发中很少用（很少自己去定义容器类，最多做框架的时候定义泛型方法），做垃圾平台（如webgl）的迁移时还要考虑性能开销。

18. 只定义刚好够用的约束条件
    - 关于 `where T` 的使用建议，基操

19. 通过运行期类型检查实现特定的泛型算法
    - 用 `is` 针对特定的类型实现更高效的方法
    - 感觉不是很常用，很少会去写容器类，更少去为了某些类型结构的性能考虑优化方案

20. 通过IComparable及IComparer定义顺序关系
    - 实现可排序或排序器，基操。我还是更喜欢用linq

21. 创建泛型类时，总是应该给实现了IDisposable的类型参数提供支持
    - 如字面意思

22. 考虑支持泛型协变与逆变
    - 如`public interface IEnumerable<out T>`的定义，容易被忽视的特性

23. 用委托要求类型参数必须提供某种方法
    - 写在构造函数中，如一些排序容器可选`IComparer<T>`作为参数

24. 如果有泛型方法，就不要再创建针对基类或接口的重载版本
    - 把 差异化/优化 操作在泛型方法内部实现

25. 如果不需要把类型参数所表示的对象设为实例字段，那么应该优先考虑创建泛型方法，而不是泛型类
    - 如字面意思，非必要就控制范围

26. 实现泛型接口的同时，还应该实现非泛型接口
    - 兼容旧api

27. 只把必备的契约定义在接口中，把其他功能留给扩展方法去实现
    - 如linq

28. 考虑通过扩展方法增强已构造类型的功能
    - 如linq，或以拓展方法的方式定义 差异化/优化 操作

### 第四章 合理地运用LINQ（Working with LINQ）

了解返回迭代器方法（懒加载方式）的利弊，以及因此而产生的`终端方法`。注意linq的返回类型和方法名称，以及闭包的原理和注意事项。

29. 优先考虑提供迭代器方法，而不要返回集合
    - 最初接触到类似的概念是python的`生成器`，是一种提供懒加载的方式。
    - 除此之外还有很多好处，见下面十几条。

30. 优先考虑通过查询语句来编写代码，而不要使用循环语句
    - 原理同29，还可以利用LINQ to SQL机制

31. 把针对序列的API设计得更加易于拼接
    - 基操，参考Linq的设计理念，连续返回`IEnumerable`

32. 将迭代逻辑与操作、谓词及函数解耦
> Decouple Iterations from Actions, Predicates, and Functions
    - 基操，参考Linq的设计理念

33. 等真正用到序列中的元素时再去生成
    - 基操，参考Linq的设计理念

34. 考虑通过函数参数来放松耦合关系
    - 例如`RemoveAll`、`Zip`和`IEquatable`、`IComparable`的区别

35. 绝对不要重载扩展方法
    - 通过声明不同的namespace会产生歧义，是这样的，可以考虑把this去掉。

36. 理解查询表达式与方法调用之间的映射关系
> Understand How Query Expressions Map to Method Calls
    - 又学了一遍sql
    - 我看这里的时候，对`SelectMany`和`Join`方法比较陌生（都是针对多个序列的方法），要记得回顾

37. 尽量采用惰性求值的方式来查询，而不要及早求值
    - 理解并认识`终端方法`

38. 考虑用lambda表达式来代替方法
    - 针对LINQ to SQL机制，关键字为表达式树和目标sql，很难理解方法在何时会难以转化成目标主机的sql，需要多加实践。

39. 不要在Func与Action中抛出异常
    - 会发生对象处于不可用状态，数据改了一半之类的抽象问题。

40. 掌握尽早执行与延迟执行之间的区别
    - 有时确实需要`ToList`立刻获取数据，而非在需要时（懒加载）

41. 不要把开销较大的资源捕获到闭包中
    - 会增加资源在内存中的驻留时间，影响性能
    - 分类讨论给解决方案，比如说在返回`IEnumerable`的方法中捕获了较大资源，就尽快调用终端方法结束查询，让资源尽快被回收。

42. 注意`IEnumerable`与`IQueryable`形式的数据源之间的区别
    - 本章经常提到一个术语“表达式树”，针对`IQueryable<T>`的拓展函数会解析“表达式树”，其作用在于：在LINQ to SQL机制下，正确的使用LINQ，可以把LINQ操作在数据库上进行，而非先传到CLR，再进行LINQ操作。
    - 兼容方法：`AsQueryable`

43. 用`Single()`及`First()`来明确地验证你对查询结果所做的假设
    - 和标题字面意思一样，还有`FirstOrDefault()`也比较常用

44. 不要修改绑定变量
    - 讨论了闭包。一个经典错误：在`for`循环里面绑定`i`，以为自己没改变量

### 第五章 合理的运用异常（Exception Practices）

在该抛异常的时候抛异常。
    - 要考虑异常的性能消耗，不要再线上环境抛无意义的异常
    - 不要用来做控制流
    - 做工具类要合理使用异常，提前发现问题

45. 考虑在方法约定遭到违背时抛出异常
    - 基操

46. 利用using与try/finally来清理资源
    - `using(null)`的作用

47. 专门针对应用程序创建异常
    - 基操，可以通过`异常转换`（即捕获某异常后抛出更详细的）明确内部异常

48. 优先考虑做出强异常保证
    - 异常的规范，“强异常保证”指抛出异常后保证所有对象都在有效状态，感觉类似sql中“事务”的概念
    - 以及一种通过封装容器，来达到安全更新容器中元素的方法

49. 考虑用异常筛选器来改写先捕获异常再重新抛出的逻辑
    - 用`when`在真正catch之前判断，而不改变异常堆栈

50. 合理利用异常筛选器的副作用来实现某些效果
    - 可以悄悄执行`when`筛选里的逻辑，而不改变异常堆栈
